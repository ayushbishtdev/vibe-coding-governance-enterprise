# AI-Generated Code Review Checklist (12 Steps, Audit-Ready)

> Enforce this checklist on every pull request containing AI-generated code (Cursor, Copilot, Cline, Windsurf, or any LLM-backed IDE). Automate Phases 1, 2, and 4 in CI/CD. The human reviewer handles Phase 3 and the Phase 4 sign-off only.

---

## How to use this checklist

- **Automated gates (Phases 1, 2, 4):** Integrate these into GitHub Actions or GitLab CI. The merge button should not activate until these pass.
- **Human gate (Phase 3):** The named reviewer does only logic validation and sign-off. Everything else is automated.
- **Evidence requirement:** Steps 11 and 12 are mandatory for SOC 2 Type II and EU AI Act Article 15 compliance. Do not skip them even for internal tooling.

---

## Phase 1 — Provenance & Authorship Validation

### Step 1 — Verify AI Provenance Tags
- [ ] Every block of AI-generated code is explicitly tagged with the model version (e.g., `gpt-4o`, `claude-3.7-sonnet`, `copilot-4`) and the original developer prompt
- [ ] Tags are present in commit metadata or a linked compliance vault entry
- [ ] No untagged code exists in the diff if AI tooling was used during the session

**Why it matters:** Without provenance tags, an incident response team cannot isolate the exact model output that caused a failure. Auditors record the inability to determine authorship as a finding.

**Automate with:** Pre-commit hooks that require `ai-author` metadata in commit messages when an IDE plugin session is active.

---

### Step 2 — Check Data Exfiltration Boundaries
- [ ] Confirm the developer did not include PII, PHI, or customer data in the prompt context window
- [ ] Confirm no proprietary algorithms, API keys, or internal architecture diagrams were pasted as context
- [ ] Confirm the AI tool used is an enterprise-ringfenced, zero-retention tenant — not a personal consumer subscription

**Why it matters:** Developers routinely feed entire codebases into the AI context window. If sent to a public endpoint, this is an immediate IP and regulatory breach. Traditional DLP tools rarely inspect IDE outbound traffic.

**Automate with:** Context window scanners or IDE policy plugins that block PII patterns before prompt submission.

---

### Step 3 — Validate Approved Tooling
- [ ] Code was generated by a tool on the approved enterprise inventory (e.g., Copilot Enterprise, Cursor with enterprise config)
- [ ] No shadow IT subscription (personal GitHub Copilot, Claude.ai free tier, etc.) was used on a corporate endpoint
- [ ] Model version is within the approved version range documented in the AI code security policy

**Why it matters:** Shadow vibe coders — developers using personal subscriptions — are one of the top five failure archetypes. The organisation loses provenance, telemetry, and contractual cover for every line generated via shadow tools.

---

## Phase 2 — Vulnerability & Hallucination Scanning

### Step 4 — Execute Prompt Injection SAST
- [ ] Specialised LLM-aware linters have run and passed
- [ ] Scan covers indirect prompt injection patterns in third-party comments, configuration files, and data sources
- [ ] No untrusted external content flows into the model's context window without sanitisation

**Why it matters:** Attackers embed malicious instructions in open-source library comments. When a developer uses Cursor to refactor that code, the embedded injection hijacks the LLM's context. The model then generates malicious code that looks natural to the reviewer.

**Automate with:** LLM SAST tools (e.g., Semgrep rules for prompt injection patterns) run as a required CI check.

---

### Step 5 — Verify Dependency Existence
- [ ] Every imported package has been cross-referenced against official registries (npm, PyPI, Maven, NuGet, etc.)
- [ ] AI-aware SBOM diff has run and returned no hallucinated packages
- [ ] No new dependency was added that does not exist on a trusted, official registry
- [ ] Typosquatting scanner has returned clean for all new packages

**Why it matters:** LLMs are predictive text engines and will confidently suggest package names that logically should exist, even if they don't. Attackers monitor LLM outputs for common hallucinations and register those fake packages. A single undetected hallucinated dependency can result in a full supply-chain compromise.

**Automate with:** AI-aware SBOM diff tools integrated into the PR pipeline as a hard gate.

---

### Step 6 — Scan for Hallucinated APIs
- [ ] All internal API endpoints referenced in AI-generated code exist in current internal API documentation
- [ ] All external API calls reference endpoints that exist in the vendor's current public documentation
- [ ] No API call targets a deprecated or removed endpoint version

**Why it matters:** AI models confidently generate calls to endpoints that do not exist or have been deprecated. These pass standard SAST and only fail at integration test or production time — at which point the blast radius is much larger.

---

### Step 7 — Assess Insecure Output Handling
- [ ] AI-generated code is treated as untrusted user input throughout the pipeline
- [ ] Proper sanitisation and validation wrappers are in place for all model outputs that interact with databases, file systems, or external services
- [ ] No AI-generated code executes external content directly without sanitisation

**Why it matters:** Traditional secure SDLC assumes a human wrote the code. AI-generated code compiles perfectly while containing flawed logic that standard SAST cannot detect. Treating AI output as trusted is equivalent to treating user input as trusted.

---

## Phase 3 — Architectural & Logic Validation (Human Reviewer)

### Step 8 — Test for Overreliance & Logic Blindspots
- [ ] The human reviewer can fully explain the algorithm generated by the AI — not just that "it looks right"
- [ ] The reviewer has traced through the logic for at least one non-happy-path scenario
- [ ] The reviewer would be able to debug this function without AI assistance if it failed in production

**Why it matters:** Developer overreliance is an OWASP LLM Top 10 vulnerability. Accepting complex AI-generated algorithms without understanding the underlying logic creates a massive technical debt and security burden. Future maintainers cannot audit or secure code they cannot understand.

---

### Step 9 — Enforce Minimum Test Coverage
- [ ] AI-generated functions are accompanied by human-reviewed unit tests (not AI-generated tests rubber-stamped without review)
- [ ] At least one test covers a failure mode or edge case the AI did not anticipate
- [ ] Integration tests cover the AI-generated component's interaction with adjacent systems

**Why it matters:** AI can generate tests as quickly as it generates code — and those tests often test only the happy path that the model itself wrote. Human-verified test coverage for failure modes is the only reliable quality gate.

---

### Step 10 — Check for Technical Debt Accrual
- [ ] The AI's solution is not overly complex or verbose relative to the problem it solves
- [ ] No deprecated methods or patterns are used in the generated code
- [ ] The code follows established architectural patterns for this codebase
- [ ] If the code introduces debt (e.g., complex boilerplate accepted for speed), it has been logged in the tech debt ledger (Tier T1–T4)

**Why it matters:** LLMs optimise for immediate syntactic correctness, not long-term maintainability. An ungoverned rework rate of 47% is the empirical baseline. Catching and logging debt at PR time is cheaper than discovering it during a Q4 sprint planning crisis.

---

## Phase 4 — Compliance & Evidence Archival

### Step 11 — Log the Review Decision
- [ ] The named human reviewer has explicitly recorded their approval in the PR
- [ ] The approval note confirms that AI-specific checks (Steps 4–10) were completed, not just standard peer review
- [ ] Reviewer identity is traceable to a named individual (not a team account)
- [ ] Timestamp of approval is automatically recorded

**Why it matters:** EU AI Act Article 26 and SOC 2 Type II require demonstrable human oversight. A rubber-stamp approval from a generic team account does not satisfy the requirement for a named, accountable individual.

---

### Step 12 — Archive the Evidence Chain
- [ ] Provenance log exported to the immutable compliance vault
- [ ] SAST scan results (Phase 2) are attached to the PR record and archived
- [ ] SBOM diff output is archived
- [ ] Human reviewer sign-off (Step 11) is archived alongside the automated results
- [ ] All artefacts are retained per the organisation's AI Act Article 12 retention policy (operational life + defined period, typically 10 years)

**Why it matters:** EU AI Act Article 12 requires retention of governance records for the operational life of the system. SOC 2 Type II reviewers test operating effectiveness over a 3–12 month observation window. Reconstructing logs after the fact is not legally accepted.

**Automate with:** Post-merge CI action that exports all gate results to an immutable audit log (e.g., S3 with Object Lock, or a dedicated GRC platform).
